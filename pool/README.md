# 账号池一致性哈希实现

## 概述

账号池现在使用一致性哈希算法来实现用户到账号的稳定路由。这确保了同一用户的请求会尽可能路由到同一个账号，从而充分利用 Claude 的缓存机制。

## 核心特性

### 1. 一致性哈希算法

- 使用 CRC32 哈希函数
- 每个账号默认创建 150 个虚拟节点
- 支持账号的动态增删，最小化映射变化

### 2. 账号亲和性

当使用 `GetByHash(key)` 方法时：
- 相同的 key（如 conversationID）总是路由到同一个账号
- 账号增删时，只有少部分用户会重新映射
- 自动跳过冷却中、额度用尽或 token 过期的账号

### 3. 降级策略

如果一致性哈希选中的账号不可用，会自动：
1. 沿着哈希环查找下一个可用账号
2. 如果所有账号都在冷却中，返回冷却时间最短的账号
3. 排除额度已用尽的账号

## 使用方式

### 基于用户/会话的路由

```go
// 使用 conversationID 或 userID 作为 key
account := pool.GetByHash(conversationID)
```

这是当前 `handler.go:445` 和 `handler.go:1041` 中使用的方式。

### 轮询方式（向后兼容）

```go
// 传统的加权轮询方式仍然可用
account := pool.GetNext()
```

## 测试结果

根据测试数据：

### 分布均匀性
- 5 个账号，1000 个用户
- 分布范围：11.1% - 29.0%
- 所有账号都被使用

### 账号增删影响
- 添加新账号：约 27% 的用户重新映射
- 移除账号：受影响账号的用户会重新映射到其他账号

### 性能
- Get 操作：O(log N)，使用二分查找
- 虚拟节点数量：150（可调整）

## 配置

虚拟节点数量在 `pool/account.go:35` 中配置：

```go
consistentHash: NewConsistentHash(150)
```

建议值：
- 100-200：适合大多数场景
- 更多虚拟节点 = 更均匀的分布，但内存占用更高

## 实现细节

### 文件结构

- `pool/consistent_hash.go` - 一致性哈希核心实现
- `pool/account.go` - 账号池管理，集成一致性哈希
- `pool/consistent_hash_test.go` - 单元测试

### 关键方法

- `GetByHash(key)` - 基于 key 获取账号（一致性哈希）
- `GetNext()` - 轮询获取账号（加权）
- `Reload()` - 重新加载账号并重建哈希环

## 优势

1. **缓存命中率提升**：同一用户的请求路由到同一账号，充分利用 Claude 的 prompt caching
2. **稳定性**：账号增删时，大部分用户的路由保持不变
3. **负载均衡**：虚拟节点确保账号间负载相对均匀
4. **容错性**：自动跳过不可用账号，支持降级策略
